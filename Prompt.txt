## 2. MISSION (what you're building, in one paragraph)

Build **SentinelNOC**: a system that (1) simulates a multi-site SD-WAN/MPLS network, (2) collects realistic telemetry from it, (3) trains models that forecast network degradation *before* thresholds are breached, (4) runs a self-hosted LLM with Retrieval-Augmented Generation (RAG) over local runbooks/incident docs to explain each forecast in plain English with a confidence score, root-cause hypothesis, affected scope, and recommended action, and (5) exposes all of this through a live dashboard. The whole pipeline must be architected to run with **zero outbound network calls** once deployed.

The system must be able to answer, live, for any current risk:
- **Q1** — What is likely to fail next, and when?
- **Q2** — Why is the risk elevated, and which signals contributed?
- **Q3** — What corrective action should be taken before impact occurs?

## 5. LLM / API CONFIGURATION — READ CAREFULLY

The deployed system must use a small, self-hosted, offline LLM (this is non-negotiable for the real submission). But your *build/dev sandbox* may not be able to run a multi-GB local model. Handle it like this:

1. Write an `LLMClient` abstraction (one Python class, one interface: `generate(prompt, context) -> structured_response`) with two backends behind a config flag (`LLM_BACKEND=local` or `LLM_BACKEND=api`):
   - `local` backend: calls a locally-served model via [Ollama](https://ollama.com)'s REST API (`http://localhost:11434`) running a small quantized model — default to `phi4-mini` or `llama3.2:3b`. Use this if Ollama is installable in your environment (try it — it's a single install script and works on CPU).
   - `api` backend: calls an LLM API using a key from an environment variable (`ANTHROPIC_API_KEY` if provided — **[ASK HUMAN]: which API key(s) will be provided, and under what env var names?** Default to treating it as `ANTHROPIC_API_KEY` and the Claude API if no answer is given). Use this only as a development/testing stand-in when Ollama truly cannot run in your sandbox.
2. Whichever backend you end up using for the demo, **say so explicitly and prominently in the README** ("This demo runs the copilot via [local Ollama / a hosted API as a sandbox stand-in for the offline model]; swapping `LLM_BACKEND=local` is the only change needed for a true air-gapped deployment.") Do not let the README imply an API-backed demo is the air-gapped deployment — that would misrepresent the project.
3. Never put a literal API key in any source file. Read it from environment variables or a `.env` file that is in `.gitignore`.
4. Build the RAG pipeline (embeddings + vector store + retrieval) to be 100% local regardless of which LLM backend is active — use `sentence-transformers` (e.g. `all-MiniLM-L6-v2`, runs on CPU) for embeddings and `ChromaDB` for the vector store. This part has no excuse to call an API.

## 6. BUILD PHASES — execute in order, run the acceptance test before moving on

### Phase 0 — Repo scaffold
Create this structure:
```
sentinelnoc/
  network_sim/         # Track A or B network + fault injection
  telemetry/           # collectors + storage
  models/              # forecasting + anomaly detection + training scripts
  rag/                 # embeddings, vector store, runbook ingestion
  copilot/             # LLMClient, prompt templates, structured-response schema
  backend/             # FastAPI app wiring it together
  dashboard/           # Streamlit (or React) UI
  scenarios/           # the 4 reference scenario scripts + results harness
  docs/                # README, architecture diagrams, DECISIONS.md, demo script
  tests/
  docker-compose.yml
  .env.example
  requirements.txt
```
**Acceptance:** `pip install -r requirements.txt` (or `docker compose build`) succeeds clean.

### Phase 1 — Network layer
Build Track A or B per Section 4: branch/hub/datacenter sites, BGP/OSPF adjacency, a notion of MPLS labels/VPN segmentation, SD-WAN-style overlay tunnels, basic QoS tagging, and a `fault_injector.py` exposing functions like `inject_congestion(link_id, ramp_minutes)`, `inject_route_flap(node_id)`, `inject_tunnel_degradation(tunnel_id)`, `inject_controller_misconfig()`.
**Acceptance:** a script that runs the topology for 5 simulated minutes, injects one fault, and prints the resulting metric drift to the console.

### Phase 2 — Telemetry pipeline
Collect/emit: interface utilization, latency, jitter, error counters (SNMP-style), routing adjacency change events (syslog-style), flow records (NetFlow-style), tunnel health stats. Normalize into one time-series store — default to a local **TimescaleDB or plain SQLite/Parquet file** (pick whichever installs cleanly in your sandbox; SQLite/Parquet is a perfectly good fallback and keeps things dependency-light).
**Acceptance:** querying the store for the last N minutes returns a clean DataFrame with no gaps, for at least 3 distinct metrics, across at least 3 nodes.

### Phase 3 — Predictive models
Build in this order, don't skip the baseline:
1. A baseline: rolling-average + trend-slope threshold detector.
2. A real forecasting model: Prophet or a small LSTM (PyTorch) per key metric (utilization, latency).
3. A graph-based check using `networkx` that correlates simultaneous anomalies across adjacent nodes (for routing-instability / cascading-failure detection).
4. A `risk_score(metric_window) -> (probability, eta_minutes)` function that combines the above.
5. A `scripts/evaluate_models.py` that runs each model against held-out fault-injection runs and computes **precision, recall, false-positive rate, and average lead time**, writing the result to `docs/model_eval_results.json`.
**Acceptance:** `model_eval_results.json` exists with real, non-placeholder numbers for at least the congestion-buildup scenario.

### Phase 4 — Knowledge base & RAG
Write 8–12 short synthetic but realistic runbook/incident documents (you write these yourself — e.g. "Runbook: Hub-Spoke Congestion," "Incident Report: BGP Flap on Branch-3, March 2025") covering the four scenario types. Embed them with `sentence-transformers`, store in `ChromaDB`. Write a `retrieve(query, k=3)` function.
**Acceptance:** a retrieval test query like "link utilization climbing on hub-spoke path" returns the correct runbook in the top result.

### Phase 5 — Offline LLM copilot
Implement the `LLMClient` per Section 5. Design a **strict structured-output prompt template** that always returns JSON with: `predicted_issue`, `confidence` (0–1), `root_cause_hypothesis`, `affected_scope` (list of node/link IDs), `recommended_action`, `time_to_impact_minutes`. Validate the JSON on every call; if parsing fails, retry once with a stricter instruction before failing loudly (never silently return malformed data).
**Acceptance:** feeding it a known risk score + retrieved runbook produces valid JSON matching the schema, grounded in the retrieved text (the `root_cause_hypothesis` should reference something actually in the runbook, not something invented).

### Phase 6 — Backend + dashboard
FastAPI backend exposing: `GET /alerts` (current prioritized risk list), `GET /alerts/{id}/explain` (the structured copilot response), `POST /chat` (free-form NL query → RAG → LLM → answer), `GET /topology` (current network graph + health). Streamlit (default) or React dashboard showing: a topology view colored by risk, a prioritized alert feed, and a chat box wired to `/chat`.
**Acceptance:** starting the backend and dashboard, injecting a fault via the Phase-1 script, and watching a new alert appear in the dashboard within the polling interval — without restarting anything.

### Phase 7 — Scenario validation (see Section 7 for exact scenarios)
Write `scenarios/run_all.py` that runs all four reference scenarios back-to-back against the live system and produces `docs/results_table.md` with columns: Scenario | Lead Time Achieved | Detection Accuracy | Copilot Explanation Correct (Y/N, judged against your own answer key) | Notes.
**Acceptance:** `results_table.md` is generated by running this script, with no manually-edited numbers.

### Phase 8 — Security/offline proof + documentation
Write `scenarios/prove_offline.py`: disables the machine's outbound network route (or runs the LLM/RAG/vector-DB stack inside a Docker network with no internet egress), runs one full alert-to-explanation cycle, and captures a `tcpdump`/packet-count log proving zero outbound packets during that cycle. Save the log to `docs/offline_proof.log`.
Then write `README.md` (architecture, how to run, results table, offline proof, what's real vs. simulated and why), `DECISIONS.md` (every fallback/decision made per Section 4–5), and `KNOWN_ISSUES.md` (anything that doesn't work, honestly).
**Acceptance:** a person who has never seen this project can clone the repo, follow the README, and reproduce the demo and the results table.

## 7. THE FOUR REFERENCE SCENARIOS (exact, use these — don't invent different ones)

1. **Progressive congestion buildup** on a hub-spoke link — utilization climbs from ~40% to >95% over 30–45 simulated minutes.
2. **BGP route flap** on one branch node, causing a downstream reroute cascade to at least one other site.
3. **Intermittent MPLS underlay failure** causing a measurable tunnel degradation (rising packet loss/jitter, not a hard down).
4. **Controller misconfiguration** causing policy drift (e.g. a QoS policy silently stops being applied to a site).

For each, the system should ideally surface a risk alert with double-digit minutes of lead time before the "official" threshold breach, and the copilot's explanation should be checkable against the runbook you wrote for it in Phase 4.
